# Выкладка в продакшен (когда проект будет готов)

Сейчас вы продолжаете работать **локально**: HTML, `admin-panel`, `crm` — без хостинга. Ниже — что уже подготовлено в репозитории и как включить базу и деплой позже.

## Что хранится где

| Данные | Локально без `DATABASE_URL` | С `DATABASE_URL` (PostgreSQL) |
|--------|---------------------------|-------------------------------|
| Пользователи, заявки CRM | `crm-db.json` | Таблица `crm_state` (один JSONB = тот же объект) |
| Контент админки, слайды, доставка | `site-content.json` | Файл на диске сервера (как сейчас) |
| Каталог (если публикуете из админки) | `products-catalog.json` | Файл на диске |
| Загрузки (аватары, фото товаров) | `uploads/` | Каталог на диске / объектное хранилище* |

\*Объектное хранилище (S3, GCS) в проекте не подключено — на VPS/Cloud Run монтируйте постоянный том для `uploads/` и JSON-файлов или вынесите медиа отдельным этапом.

## Переменные окружения

Скопируйте `.env.example` → `.env` на сервере и задайте как минимум:

- `NODE_ENV=production`
- `JWT_SECRET` — случайная строка **не короче 32 символов**
- `ALLOWED_ORIGINS` — ваши `https://домен` через запятую
- `DATABASE_URL` — строка подключения к PostgreSQL (например Cloud SQL)
- при необходимости `PGSSLMODE=require`

## PostgreSQL локально (проверка до хостинга)

```bash
npm run docker:up
# В .env:
# DATABASE_URL=postgresql://dp:dp_local_dev@localhost:5432/coatings
npm run crm:import-json   # один раз: перенос crm-db.json → БД
npm start
```

Проверка: `GET /api/health` → `"crmStore": "postgresql"`.

Вернуться к файлу: уберите `DATABASE_URL` из `.env` и перезапустите сервер.

## Google Cloud SQL

1. Создайте инстанс PostgreSQL, пользователя и БД.
2. Включите подключение с вашего бэкенда (VPC, Cloud SQL Auth Proxy или публичный IP с ограничением по IP).
3. Пропишите `DATABASE_URL` и `PGSSLMODE=require` в секретах Cloud Run / в `.env` на VM.
4. Один раз выполните импорт: с машины с доступом к БД запустите `npm run crm:import-json` с тем же `DATABASE_URL`, либо дайте серверу создать пустую строку при первом старте и импортируйте позже.

Схема таблицы дублируется в `deploy/postgres-crm-state.sql`; при старте `server.js` таблица создаётся сама.

## Docker-образ

```bash
docker build -t dp-coatings-api .
docker run --env-file .env -p 3000:3000 dp-coatings-api
```

Убедитесь, что к контейнеру примонтированы или сохраняются в томе: `uploads/`, `site-content.json`, `products-catalog.json`, `analytics-store.json` (если используете аналитику на файле). Иначе после перезапуска контейнера правки пропадут.

## Правки сайта, админки и CRM после выкладки

Рабочий процесс не меняется: правите HTML/JS/CSS в репозитории, деплойте новую версию статики и/или образа API. Структура CRM в коде по-прежнему опирается на один JSON-объект (`users` / `leads` / `meta`), поэтому новые поля в заявках или профиле **не требуют SQL-миграций**, пока вы укладываетесь в этот снимок.
